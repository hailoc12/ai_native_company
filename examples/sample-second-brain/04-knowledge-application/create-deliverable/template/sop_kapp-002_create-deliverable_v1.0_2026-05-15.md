## 1. Tổng Quan

Biến tri thức từ vault thành deliverable cụ thể, hữu hình — là proof rằng knowledge đã được áp dụng, không chỉ được lưu trữ.

---

## 2. Deliverable Types & Approach

| Deliverable Type | Knowledge Input | Primary Tool | Time |
|-----------------|----------------|-------------|------|
| Written report | Gems + synthesis notes | Claude AI + template | 2-4h |
| Presentation/Slide | Top 5 gems + examples | Slide template | 1-2h |
| Article/Post | 1-3 gems + personal experience | Claude AI + editor | 1-3h |
| Decision brief | Relevant gems + analysis | Decision template | 1-2h |
| Action plan | Process gems + project context | OmniFocus + template | 30-60 min |
| Ebook/Course | Full synthesis cluster | vibe-ebook-orchestrator | Multiple days |

---

## 3. Quy Trình

| # | Hành động | Chi tiết | Output |
|---|----------|----------|--------|
| 1 | Select deliverable type | Based on audience + purpose | Type selected |
| 2 | Outline from vault | Map gems/notes to outline sections | Draft outline |
| 3 | Generate first draft | Claude AI + relevant vault notes | First draft |
| 4 | Enrich with personal context | Add experience, examples, opinions | Enriched draft |
| 5 | Quality review | vibe-review or self-review checklist | Reviewed draft |
| 6 | Finalize and export | Format for audience | Final deliverable |
| 7 | Archive knowledge used | Log: which gems contributed to what | Attribution log |

---

## 4. Checklist
- [ ] Deliverable clearly references at least 2 gems/insights from vault?
- [ ] Written for specific audience (not generic)?
- [ ] Quality reviewed?
- [ ] Knowledge attribution logged?
